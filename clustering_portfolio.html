Project overview

What this project does

Where you live shapes what surrounds you — but does the same kind of city feel the same regardless of which country it's in? This project uses Foursquare venue data to answer that question, clustering UK and Spanish cities by their venue fingerprint rather than their geography.

Research question

Do UK and Spanish cities cluster by country — or by city type? Does the dominant venue category (cultural, gastronomic, outdoor) override national geography as a grouping signal?

Key methodological decision

Using mean frequency rather than raw venue counts to represent each city normalises for the fact that larger cities return more total venues. A city with 98 venues and one with 40 are compared proportionally, not by absolute volume.

No labelled training data

K-Means discovers cluster structure from the venue frequency matrix alone — no pre-assigned city type labels, no demographic variables. The cluster character emerges entirely from the first most common venue type per group.

Methodology

How it was built

Five stages transform a city coordinate file into a clustered, colour-coded interactive map. Each decision is documented with its analytical rationale.

01

Data collection

Foursquare API — getVenues()

A custom getVenues() function loops through every city in the CSV, calls the Foursquare Explore endpoint, and returns the name, coordinates, and category of up to 100 nearby venues. Results are collected into a flat list and assembled into a single venues DataFrame.

⚠ Note: Foursquare Places API v2 (legacy) is deprecated. The endpoint and CLIENT_ID/CLIENT_SECRET auth model no longer work with new credentials. Migrate to Foursquare Places API v3 for live re-runs.
02

Feature engineering

One-hot encode → group by city → mean frequency

pd.get_dummies() expands the Venue Category column into binary indicator columns — one per unique category. Rows are then grouped by city and the mean of each binary column is computed, giving each city a fractional venue profile.

✦ Mean frequency normalises for venue count differences between cities. Without this step, large cities would dominate the clustering simply because they return more venues — not because their profile is more extreme.
03

Venue ranking

Top 10 most common venues per city

A helper function return_most_common_venues() sorts each city's feature vector in descending frequency order and extracts the top 10 venue type labels. This creates a ranked venue table — the primary human-readable output used to interpret what each cluster actually represents.

✦ The 1st Most Common Venue becomes the de-facto cluster label. Inspecting its value_counts() per cluster is faster and more interpretable than examining centroid coordinates in high-dimensional space.
04

Clustering

K-Means (k=8, random_state=1)

Scikit-learn KMeans is fit to the city venue frequency matrix with the city column dropped. random_state=1 ensures reproducibility. Labels are joined back to the cities DataFrame to carry coordinates for mapping.

⚠ Note: k=8 is a deliberate choice, not data-derived. No elbow curve or silhouette analysis is performed. A natural extension is to plot inertia across k=2 to 15 and select the inflection point empirically.
05

Interpretation

Cluster composition — cities and venue signatures

After fitting, two analysis passes run: first, the count of cities per cluster (value_counts on labels); second, the most common 1st venue type per cluster (value_counts on the ranked venue column filtered by cluster label). A grouped display lists every city in each cluster.

✦ Cross-referencing cluster membership with country reveals whether UK and Spanish cities mix within clusters — a direct answer to the core research question without any additional statistical test.
06

Targeted analysis

River venue filter — category as geographic signal

A second focused analysis drops all columns except River and city, merges coordinates back, and filters to cities where River frequency is non-zero. This demonstrates how a single venue category column can act as a geographic filter — surfacing cities with active riverfront activity in Foursquare data.

✦ This pattern generalises: any venue category column can become a filter. "Cities where Museum frequency > 0.1" or "Cities where Beach frequency > 0.05" would answer tourism-relevant questions with no additional data collection.

Key insights & findings

What the data reveals

These analytical conclusions emerge from the cluster structure — each one follows from a specific stage of the notebook and leads to a concrete follow-up question.

1

City type outweighs national geography — UK and Spanish cities mix within clusters

The K-Means clusters do not simply separate British from Spanish cities. Instead, cities with similar venue compositions — a coastal resort in the UK and a beach town in Spain, or a cultural capital from either country — cluster together. This directly answers the research question: the venue fingerprint of a city is a stronger grouping signal than which country it sits in. → Venue profile is a better proxy for "city character" than national origin.

2

The 1st Most Common Venue is a reliable cluster label — no manual annotation needed

Inspecting cluster_n['1st Most Common Venue'].value_counts().head() per cluster produces interpretable human-readable names without any supervised labelling. A cluster where Restaurant dominates tells a different story from one where Gym or Museum leads. This is the unsupervised equivalent of feature importance — the top venue is the defining dimension for each group. → Value_counts() on the ranked venue column is all the label generation you need.

3

The River filter demonstrates that venue columns are reusable geographic queries

Filtering to cities where River frequency is non-zero produces a sub-map that is not simply "all cities near water" — it is specifically cities where Foursquare users have checked into river-category venues within 5 km of the centre. The same logic applies to any category: Museum, Beach, Hiking Trail. A single line of filtering code turns any venue column into an answerable geographic question. → The feature matrix doubles as a reusable query engine for place-type analysis.

4

Venue density bias is the most important data quality issue in this pipeline

The Foursquare API returns up to 100 venues per city — but smaller cities may only have 20–30 venues logged, while London or Madrid saturate the limit. Mean frequency normalises for this to some extent, but a city with 30 venues will still have noisier category proportions than one with 98. Clusters containing predominantly small cities may reflect data sparsity as much as genuine venue similarity. → Always check total venue count alongside cluster membership before interpreting a cluster as analytically meaningful.

5

8 clusters provides useful granularity — but the right k is an open question

The choice of k=8 produces clusters that are interpretable and varied — there is no single dominant mega-cluster absorbing most cities. However, without an elbow curve or silhouette score, k=8 cannot be claimed as optimal. It is possible that k=5 or k=6 would produce equally coherent clusters with less fragmentation of small groups. The current k is a starting point for exploration, not a definitive answer. → Run inertia across k=2 to 15; the elbow point is the more defensible choice for a published analysis.

Further work

What I'd build next

Three directions that would sharpen and extend these findings — each addresses a known limitation of the current pipeline.

📐

Elbow & silhouette analysis to find the optimal k

k=8 was chosen by intuition. Plotting inertia across k=2 to 15 and overlaying silhouette scores would give a data-derived justification for the cluster count — or reveal that a different k produces tighter, more separated groups. DBSCAN is also worth testing: it does not require k to be specified and can identify outlier cities that do not fit any cluster cleanly.

🌍

Migrate to Foursquare Places API v3 and extend to more countries

The v2 API used here is deprecated — live re-runs require migration to Foursquare's current API. Once restored, extending to France, Germany, and Italy would test whether the cluster structure generalises beyond the UK/Spain pair. The pipeline needs only a new CSV of cities with coordinates and the same getVenues() logic updated for the v3 endpoint format.

📊

Add demographic context to cluster profiles

Venue frequency tells you what a city has — but not why. Merging cluster membership with population size, GDP per capita, or tourism visitor numbers would test whether clusters correspond to economic categories as well as venue ones. A cluster of "cultural capital" cities should show elevated museum density and high tourism. A cluster of "dormitory towns" should show high supermarket and commuter-transport density with low leisure venues.

Limitations

Caveats worth knowing

📡 Foursquare API v2 is deprecated

The data collection stage of this notebook uses the legacy Foursquare Places API v2, which has been deprecated. The CLIENT_ID / CLIENT_SECRET auth model and the /v2/venues/explore endpoint no longer work with new credentials. The notebook reflects the original implementation; a v3 migration is required for live re-runs.

🎯 k=8 is not data-derived

The cluster count is set manually without elbow curve or silhouette analysis. Different values of k would produce different cluster structures — some potentially more coherent than k=8. Any interpretation of the 8 clusters as "the right grouping" should be treated as exploratory rather than definitive until a systematic k selection method is applied.

📉 Venue density varies by city size

Larger cities approach the 100-venue API limit; smaller towns may return 20–30. Mean frequency partially normalises for this, but sparse cities have noisier profiles. A cluster dominated by small cities may reflect data incompleteness as much as genuine venue similarity. Checking total venue count per city alongside cluster membership is good practice before drawing conclusions.
