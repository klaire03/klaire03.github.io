Project overview

What this project does

Weather is a sequence problem. Clothing is a language problem. This project solves both — by building a time-series LSTM that forecasts the next 12 hours of conditions, then a semantic recommender that converts those numbers into natural language and matches it to the right outfit using sentence embeddings.

Research questions

Three analytical questions

Can an LSTM accurately forecast short-range weather from 14 days of hourly data? Does converting numeric conditions to natural language produce better clothing matches than direct feature comparison? How much recommendation quality is lost by random sampling vs exhaustive search?

Key methodological decision

Language as the bridge

Using natural language as the bridge between weather numbers and clothing descriptions — rather than a direct numeric feature match — is the core design choice. It leverages the semantic knowledge already baked into the sentence transformer's pre-training, without any labelled weather-clothing training data.

No labelled training data

Zero labelled pairs required

The clothing recommender requires zero labelled weather–outfit pairs. The sentence transformer's pre-trained knowledge of language handles the association. Clothing embeddings are pre-computed and cached — only the weather sentence needs encoding at request time.

Live app — select city, get forecast & outfit

The deployed FastAPI app accepts a city selection and returns the full pipeline output in a single view: live weather conditions, a Leaflet.js map, a 12-hour LSTM temperature forecast chart, and a categorised outfit recommendation grid with product images pulled from the clothing catalogue. Each recommendation includes category, item name, and a confidence score derived from cosine similarity.

Methodology

How it was built

Six analytical stages run in sequence — from raw API ingestion through to a ranked outfit recommendation list. Each technique was chosen to answer a specific question, and each decision is documented with the rationale including where a simpler approach was deliberately rejected.

01

Data ingestion

🌦️ Open-Meteo API + local CSV cache

Hourly weather history is fetched via openmeteo_requests with a 5-retry session. Results are stored as location_startdate_enddate.csv. On every load the column schema is validated against the expected variable list — if it mismatches, the cache is deleted and refetched. Eight variables are collected: temperature, humidity, dew point, apparent temperature, rain, wind speed, cloud cover, and pressure. Response is unpacked variable-by-variable using hourly.Variables(idx).ValuesAsNumpy() and assembled into a DataFrame with a generated pd.date_range index.

⚠ Note: Only 4 cities are currently supported via a hardcoded coordinate dictionary. The get_coords_from_location_name_dummy() function must be extended or replaced with a live geocoding call to support arbitrary locations. Any unknown city name returns None and fails silently downstream.
02

Feature engineering

🔁 Missing timestamp interpolation + cyclical time encoding

Missing hourly rows are detected by building a complete pd.date_range and diffing against the actual index. Gaps are filled with .interpolate(method='linear'). The datetime column is then encoded into six cyclical features — sin/cos for hour of day (÷24), day of week (÷7), and day of year (÷365) — and dropped. Three temporal dimensions encoded as sin/cos pairs produce 6 new columns that preserve the circular nature of time.

✦ Cyclical encoding prevents the LSTM from treating time as linear. Without it, the model sees hour 23 and hour 0 as maximally distant — the opposite of their true relationship. Sin/cos encoding maps both to adjacent points on a circle. This single feature engineering decision has more impact on LSTM generalisation than architecture tuning.
03

LSTM forecasting

🧠 336-hour input window → 12-hour multi-target prediction

The preprocessed array is shaped to (1, 336, n_features) via np.expand_dims and passed to the loaded Keras model. The model outputs a (12, 4) array — 12 hours of temperature, humidity, rain, and wind. A saved joblib scaler reverses the MinMax normalisation before the values are returned. Negative predictions are clipped to zero. The model and a separate target scaler are versioned by timestamp and loaded at inference by globbing the most recent checkpoint in /models/.

✦ A separate target scaler (covering only the 4 output variables) is saved alongside the model. Using the full input scaler would attempt to inverse-transform variables the model never predicts, producing incorrect output shapes.
04

Weather narration

💬 Numeric forecast → natural language description

The 12-hour forecast is aggregated to a single representative condition: median temperature, median humidity, maximum wind, maximum rain. A describe_by_threshold(value, thresholds) function scans ordered (limit, description) pairs and returns the first matching descriptor. Four independent lookup tables cover temperature, humidity, wind, and rain. The outputs are assembled into a complete sentence: "At around 14 o'clock the air is cool and fresh at 12°C, it is slightly breezy with a windspeed of 8 km/h..."

⚠ Note: The threshold values are business rules, not data-derived. Different threshold tables would produce different sentence outputs and therefore different recommendation results — they are the primary lever for tuning recommendation behaviour without retraining any model.
✦ Threshold tables are tunable without retraining — they define what "cold enough for a coat" means. This makes recommendation quality partly a product design problem, not just a machine learning problem.
05

Clothing recommendation

👗 Sentence embedding + cosine similarity over pre-computed clothing vectors

The weather sentence is encoded with all-MiniLM-L6-v2. Clothing descriptions are encoded once into 384-dimensional vectors and saved as embeddings_{category}.npy per category. At inference, only the weather sentence is encoded live. For each clothing category, up to 200 items are sampled and their pre-computed embeddings are retrieved. cosine_similarity([weather_emb], clothes_emb_sample) scores each item. The top 5 are returned; product image links are validated with an HTTP HEAD request and dead links are excluded. Cosine similarity is preferred over Euclidean distance — it measures the angle between vectors, capturing semantic direction rather than magnitude.

✦ Pre-computing clothing embeddings at catalogue load time reduces inference latency by ~95%. The sentence transformer only needs to encode a single weather sentence per request — not the entire clothing catalogue.
⚠ Note: Random sampling of 200 items per category introduces non-determinism — results vary between calls on large catalogues. Consider FAISS approximate nearest-neighbour search for deterministic, scalable retrieval at production catalogue sizes. Link validation adds 1–2 seconds per recommendation call due to sequential HTTP requests.

Outfit recommendation output — top 5 per category with cosine similarity confidence scores.

06

Deployment

🚀 FastAPI → Docker → Google Cloud Run

The full pipeline — weather fetch, LSTM inference, recommendation — is served via a uvicorn FastAPI application. The Docker image bakes in the trained model, scaler, clothing CSVs, and pre-computed embeddings. Cloud Run deployment is automated via make docker_push && make docker_deploy. The image targets linux/amd64 for GCP compatibility. The model is cached locally in /models/all-MiniLM-L6-v2/ to avoid download latency in production.

✦ Baking the model and data into the Docker image avoids cold-start latency from remote model fetches. The tradeoff is a larger image size — acceptable for a single-model API but worth revisiting if multiple model versions need to coexist.

FastAPI app — city selection returns live weather, Leaflet.js map, LSTM forecast chart, and outfit recommendation grid.

# End-to-end pipeline — API ingestion through to ranked clothing recommendations
1. Install & import — openmeteo_requests, keras, sentence_transformers, fastapi, sklearn, joblib, pandas, numpy
2. Geocode city — get_coords_from_location_name_dummy(location) → lat, lon, timezone
3. Fetch weather history — Open-Meteo archive API; 8 hourly variables; start_date = today - 14 days
4. Cache to CSV — location_startdate_enddate.csv; schema-validated on load; delete & refetch on mismatch
5. Clean timestamps — pd.date_range diff; .reindex(full_index).interpolate(method='linear')
6. Timezone filter — tz_localize(UTC) → tz_convert(local); filter df to rows ≤ now; take last 336 rows
7. Cyclical encoding — sin/cos for hour (÷24), day_of_week (÷7), day_of_year (÷365); drop date column
8. Shape input — np.expand_dims(df, axis=0) → (1, 336, n_features) batch tensor
9. Load LSTM model — glob latest weather_predictor_{location}*.keras; load_model with custom MSE
10. Load scaler — glob latest weather_y_scaler_{location}*; joblib.load
11. Predict — model.predict(X) → (1,12,4); squeeze; scaler.inverse_transform; clip(lower=0)
12. Build forecast DataFrame — columns: time, temperature, humidity, wind, rain; 12 rows
13. Aggregate forecast — median(temp), median(humidity), max(wind), max(rain), first timestamp
14. Narrate weather — threshold lookup tables → qualitative descriptors → assembled sentence string
15. Load clothing data — accessories.csv, shoes.csv, top.csv, bottom.csv → pd.concat; load .npy embeddings per category
16. Encode weather — SentenceTransformer('all-MiniLM-L6-v2').encode(weather_sentence) → 384-dim vector
17. Sample & score — per category: sample 200 items; cosine_similarity([weather_emb], clothes_emb) → scores array
18. Rank & validate links — argsort(scores)[::-1][:5]; HTTP HEAD request per image link; dead links skipped
19. Serve via FastAPI — uvicorn api.fast:app --host 0.0.0.0 --port $PORT; Docker + Cloud Run

Key insights & findings

What the pipeline reveals

These analytical conclusions emerge from building and evaluating the system — each one surfaces a specific design decision and points toward a concrete next question.

1

💬 Natural language is a better bridge than feature vectors — zero labelled training data required

The most counterintuitive design decision is deliberately not building a direct numeric-to-clothing model. Instead, the weather forecast is converted to natural language, then matched semantically. This works because all-MiniLM-L6-v2 was pre-trained on a vast corpus that includes product descriptions, weather language, and everyday clothing vocabulary. The semantic associations between "cold and rainy" and "waterproof jacket" are already encoded in its weights.

→ Pre-trained language models eliminate the cold-start problem for recommendation: zero labelled weather–outfit pairs are required.

2

🔁 Cyclical encoding is not optional — it determines whether the LSTM can generalise across the year

Without sin/cos encoding, the LSTM treats time as a linear variable: hour 23 appears maximally distant from hour 0, and December 31 is treated as completely unlike January 1. A model trained this way will consistently misforecast weather at day and year boundaries — precisely the transitions where forecast accuracy matters most. Cyclical encoding maps all time dimensions onto circles, preserving their true topology.

→ A single feature engineering decision — how time is represented — has more impact on LSTM forecast quality than architecture tuning.

3

🎛️ Threshold tables are the primary lever for recommendation quality — and they're not learned from data

The qualitative descriptors ("cool and fresh", "slightly breezy") are produced by hardcoded threshold tables, not by any model. Changing where the temperature boundary between "fresh" and "mild" sits changes which clothing items get recommended — without touching the LSTM or the sentence transformer. This means recommendation quality is partly a product design problem, not just a machine learning problem.

→ Tuning threshold tables is the fastest, cheapest, and most interpretable way to improve recommendation relevance.

4

⚡ Pre-computing clothing embeddings reduces inference latency by ~95% — the bottleneck shifts to link validation

Encoding the full clothing catalogue at request time would take several seconds per category. Pre-computing and caching embeddings as .npy files means the transformer only encodes one sentence at inference. After this optimisation, the dominant latency is sequential HTTP HEAD requests validating product image links — one per candidate item. A link health cache refreshed on a schedule would eliminate this bottleneck entirely.

→ The embedding cache is the right architectural decision; async link validation is the missing companion optimisation.

5

📍 4-city hardcoding is the system's most critical scalability constraint

The city coordinate dictionary in get_coords_from_location_name_dummy() silently returns None for any unsupported location — a failure mode that propagates invisibly through the pipeline until it crashes at the API call. The real geocoding function using geopy.Nominatim already exists in the codebase; the dummy version was introduced for reliability during development and was never removed.

→ One function swap turns a 4-city demo into a global product.

Further work

What I'd build next

Three directions that would materially sharpen the system — each targets a known limitation of the current architecture.

🌍

Replace hardcoded cities with live geocoding

The geopy.Nominatim geocoder already exists in the codebase alongside the dummy dictionary. Swapping get_coords_from_location_name_dummy() for the live version would make the pipeline work for any city on earth. The Open-Meteo API has global coverage; the only engineering work is adding a fallback for geocoding failures and caching coordinates to avoid repeated API calls.

⚡

FAISS nearest-neighbour search — deterministic, scalable clothing retrieval

The current random 200-item sample introduces non-determinism and misses high-scoring items that fall outside the sample window. A FAISS index built once at catalogue load time would retrieve the true top-k items from the full catalogue in milliseconds — no sampling, no variability, and no performance penalty. This becomes critical as the clothing catalogue grows beyond a few thousand items.

🔁

Scheduled model retraining — keep the LSTM accurate as patterns shift

The LSTM is trained on a historical snapshot and never updated. Weather patterns shift seasonally and year-over-year; a model trained in summer will gradually degrade on winter forecasts if left static. A monthly retraining job pulling fresh data from Open-Meteo and redeploying the updated checkpoint via the existing Docker pipeline would maintain forecast accuracy without architecture changes.

Limitations

Caveats worth knowing

📍 Only 4 cities are supported

Location lookup is hardcoded to Berlin, Marseille, Porto, and London. Any other city name silently returns None and fails downstream. The real geocoding function using geopy.Nominatim is already in the codebase — the dummy version was a reliability shortcut during development and was never removed.

🎲 Recommendation results are non-deterministic

Each call samples up to 200 clothing items per category at random. On catalogues larger than 200 items per category, the top-ranked results may differ between requests. This is a deliberate speed tradeoff — cosine similarity over the full catalogue without sampling is slow at inference time — but it means results are not reproducible without fixing the random seed.

🔗 Dead product links are excluded, not repaired

Image links are validated with an HTTP HEAD request and silently skipped if they return a 4xx or 5xx status. This means the number of recommendations returned can fall below top_k=5 if many links in the top results are broken. The clothing catalogue needs periodic link-health maintenance to prevent recommendation quality degrading over time.
