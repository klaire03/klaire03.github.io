Project overview

Why this project exists

The goal was to simulate a production-grade churn analytics framework for a SaaS cybersecurity product — moving from raw operational data through descriptive reporting to a predictive system that quantifies revenue at risk. Three questions framed every decision: what behaviour predicts churn, who is at risk today, and how much revenue is exposed?

The data

Five relational tables — accounts, subscriptions, feature_usage, churn_events, and support_tickets — connected through foreign keys but full of inconsistent date formats, text-encoded booleans, and unnormalised flags.

Key design decision

KPI definitions — churn rate, escalation %, tenure buckets, risk tiers — are standardised in the SQL mart and Tableau calculated fields first. This ensures the dashboard and the ML model are always measuring the same thing.

Why PR-AUC, not ROC-AUC

At 22% churn the dataset is moderately imbalanced. ROC-AUC can look good by exploiting the majority class. PR-AUC measures precision and recall on the minority class — the churners we actually need to find.

Methodology

How it was built

Three phases in sequence — SQL data engineering, Tableau BI, and Python ML. Each phase is documented with the specific decisions made and why, including where a simpler approach was deliberately tried first.

Phase 1 · Data Engineering

01

Staging layer

🧹 One clean view per source table

Each staging view solved the same categories of problems: dates parsed with STR_TO_DATE, text flags like 'true'/'false' converted to 0/1 integers via CASE WHEN, categorical fields normalised, referential integrity confirmed. Four staging views — stg_accounts, stg_subscriptions, stg_support, stg_feature_usage — so every downstream query could trust its inputs without defensive casting.

✦ Centralising data cleaning in staging views means a fix in one place propagates everywhere. If a date format changes upstream, one view update corrects all dashboards and the ML mart simultaneously.
02

Aggregation layer

📊 One row per account — three aggregation views

Three views collapsed each domain to one row per account: agg_account_revenue (total MRR/ARR, upgrade/downgrade/auto-renew flags), agg_account_usage (distinct features used, total usage time, total errors, average errors per event), and agg_account_support (ticket count, average resolution time, average first response, average CSAT, escalation count). These lifetime signals feed both Tableau and the ML model.

✦ Aggregating to account level in SQL before Tableau eliminates row-level duplication in joins. Tableau's LOD expressions can do this — but doing it upstream is faster, cleaner, and keeps the mart reusable outside Tableau.
03

Mart layer

🏗️ mart_account_health — single analytical row per account

A final mart view joins all aggregation views to produce one trusted, denormalised row per account: account profile, revenue metrics, usage signals, support history, and churn flag. This is the single source of truth — the mart feeds both the Tableau workbook and the Python ML pipeline, guaranteeing both are working from identical feature values.

✦ A shared mart means dashboard numbers and model scores will always be consistent. If the Tableau churn rate is 22%, the model is trained on the same 22%. Divergence between reporting and modelling is a common production failure this design explicitly prevents.
04

Rule-based baseline

⚖️ mart_account_risk — SQL risk scoring model

Before progressing to ML, a rule-based scoring view assigned point weights to behavioural flags: low usage (+2), high error count (+2), support escalations (+2), low CSAT (+2), downgrade history (+3), auto-renew disabled (+3). Accounts were segmented into High (≥8), Medium (≥4), and Low (<4) risk tiers. Validating these tiers against actual churn rates revealed limited separation between segments — a useful negative result that provided the data-driven justification for building the ML model.

⚠ Note: The rule-based model's limited separation isn't a failure — it's the expected finding. Churn is rarely caused by one signal in isolation. Weighted rules can't capture the interaction between low usage AND downgrade history AND disabled auto-renew. That's precisely what XGBoost is built to find.

Phase 2 · Business Intelligence

05

Executive dashboard

📈 Leadership-level visibility

Acquisition momentum, ARR by plan tier and industry, and a global user distribution map. Built to answer the top-line question: where is the business growing and where is the revenue concentrated? KPI definitions — churn rate, tenure buckets, plan movement — were standardised as Tableau calculated fields so all three dashboards use the same logic.

06

Retention dashboard

🔴 Customer retention & risk

Dedicated churn intelligence — 22% overall churn rate, churn by tenure cohort, plan tier movement, churn reason breakdown, and an at-risk accounts table giving CS teams an account-level intervention queue ranked by SQL churn risk score. The plan movement calculated field distinguishes "Upgraded from Basic", "Downgraded from Pro", and "No Change" — making contraction visible before it becomes churn.

07

Product & support dashboard

🛠️ Product usage & support performance — the churn signal dashboard

Operational metrics linking product experience and support quality to retention. The headline figures — 87-hour average first response and 19% escalation rate — both emerged as significant churn correlates in the XGBoost feature importance analysis, confirming this dashboard wasn't just measuring satisfaction: it was measuring churn risk. CSAT by plan tier surfaces whether support quality varies across account value segments. Feature adoption rates show which capabilities drive engagement and which are underused. Calculated fields: escalation rate as a formatted percentage; resolution time formatted as "35h 39m"; risk tier derived from the mart's churn_risk_score.

⚠ Note: The 87-hour first response time isn't just bad for CSAT — it ranked as a direct churn predictor in the ML model. Operational dashboard metrics and ML feature importance converging on the same signals increases confidence in both.

Model 02 · Intermediate

Random Forest

0.623

ROC-AUC

Captures non-linear feature interactions logistic regression misses. Churn rarely emerges from one factor — it typically combines low usage with downgrade history. n_estimators=300, class_weight="balanced". Also provides feature importance as a diagnostic for the SHAP analysis.

n=300 estimators

Model 03 · Production

XGBoost

Best

PR-AUC · primary metric

Gradient boosting reduces both bias and variance simultaneously. Tuned for class imbalance via scale_pos_weight = n_negative / n_positive. Delivered the strongest precision-recall balance — best suited when minority-class (churner) performance is the critical metric. max_depth=4, learning_rate=0.05, subsample=0.8.

Production model

SHAP explainability: Feature importance reveals the population-level churn drivers — auto-renew status, downgrade history, error frequency, support escalations. SHAP values go further, quantifying each feature's contribution to each individual prediction: "This account's 72% churn risk is primarily driven by downgrade history and disabled auto-renew." SHAP transforms a black-box score into an actionable narrative. A customer success team can respond to a reason, not just a number.

Key findings & recommendations

What the data says

Two themes emerged: the operational and behavioural drivers of churn, and the commercial picture that determines where retention effort creates the most revenue value.

1

📉 22% churn — retention is the primary growth lever at current acquisition rates

With acquisition steady at 15–30 signups per period and not accelerating, the compounding effect of reducing churn outweighs new logo acquisition. Every retained account is revenue that doesn't need replacing. Modelling a 2–3 percentage point reduction in churn shows greater NPV than the equivalent spend on acquisition at current CAC.

→ Model the revenue impact of a 2–3 pp churn reduction before committing further acquisition budget.

2

📅 Churn rises with tenure: 18% → 27% in the 24–50 month window

Longer-tenured accounts churn at higher rates — a signal of weak long-term value expansion. The product delivers initial value but appears to struggle growing with customer needs past the two-year mark. This is the inverse of the typical SaaS pattern where tenure reduces churn risk as switching costs accumulate.

→ Introduce mid-lifecycle health checks and expansion playbooks for accounts approaching the 24-month mark.

3

⬇️ Pro downgrades (36%) signal contraction before churn

Downgrades are a leading indicator. The 36% downgrade rate among Pro accounts points to value perception issues — customers are contracting before they leave entirely, giving a clear intervention window. This is also the feature with the third-highest SHAP importance in the XGBoost model, confirming it's a genuine signal and not noise.

→ Trigger proactive outreach and a value review immediately after any downgrade event, not at renewal.

4

💰 Churn is partly economic — pricing ranks as a top stated reason

Alongside support friction and product gaps, pricing and budget feature prominently among churn reason codes. This means retention strategy needs commercial flexibility, not just product or service improvements. The XGBoost model's strong performance on these accounts suggests pricing sensitivity has a distinct behavioural signature detectable before the stated reason is given.

→ Test flexible pricing, annual commitment incentives, or tailored packaging for price-sensitive at-risk accounts.

5

🔧 87h first response and 19% escalation rate are direct churn signals

Support failure isn't just a satisfaction issue — both metrics ranked as significant churn drivers in XGBoost feature importance, confirming what the Product & Support dashboard surfaced. The convergence of dashboard operational metrics and ML feature importance on the same signals increases confidence in both. Operational friction translates directly to customer exits, especially for high-MRR accounts.

→ Set tighter SLA targets for high-MRR accounts and implement escalation monitoring with automated senior routing.

6

💎 Revenue-weighted prioritisation is non-negotiable

The scoring table proves the point directly: A-d4e004 has a Low churn probability (0.163) but ranks 4th by actual revenue exposure (£22,535) because its MRR is £138,060 — above three High-risk accounts. A CS team working from a probability-sorted queue would never reach it. The FinTech vertical, which generates the highest ARR, requires dedicated monitoring for exactly this reason.

→ Sort the intervention queue by P(Churn) × MRR, not by churn probability alone.

7

🌍 France is the strongest geographic churn predictor

Country emerged as a significant feature in the XGBoost model, with France ranking as the top geographic churn predictor. This points to localised pricing, competitive, or localisation factors beyond what global product and support improvements would address. It also suggests the model has learned something the dashboard aggregate figures don't surface — geographic segmentation of churn is underexplored in the current BI layer.

→ Conduct a France regional deep-dive: competitive benchmarking, pricing analysis, and localisation gaps.

Further work

What I'd build next

Three directions that would sharpen and extend these findings — each addressing a known limitation of the current pipeline.

🔁

Churn prediction at the event level

The current model operates on lifetime account aggregates. Rebuilding it on time-windowed features — rolling 90-day usage, recent ticket velocity, login cadence — would detect deterioration as it happens rather than retrospectively. An account that used to log in daily but hasn't in three weeks is at risk in a way the lifetime average misses entirely.

💰

Revenue simulation model

Model the MRR impact of reducing churn by 1, 2, and 5 percentage points under different acquisition scenarios. This turns the analysis from descriptive to financial planning — and makes the business case for retention spend concrete. The data and the model are already in place; a simulation layer is the natural next step.

🧪

A/B test the intervention hypotheses

The recommendation to trigger outreach after downgrades and improve SLA response for high-MRR accounts is directional. Running controlled tests on these interventions would validate whether the model's feature signals translate to actual retention lift — closing the loop between prediction and measurable outcome.

Limitations

Caveats worth knowing

📊 Simulated dataset

The underlying data is synthetic — designed to reflect realistic SaaS patterns but not derived from a live product. The modelling approach and analytical decisions are production-grade; the specific numbers are illustrative. In a live deployment, all thresholds and weights would require calibration against real churn outcomes.

⚖️ Class imbalance handled, not eliminated

At 22% churn the dataset is moderately imbalanced. scale_pos_weight and PR-AUC prioritisation address this, but real-world deployment would require threshold calibration against actual cost-of-misclassification data — the cost of a false negative (missed churner) is rarely equal to a false positive (unnecessary outreach).

💬 Stated reasons vs. behavioural signals

Churn reason codes are self-reported and often imprecise. Behavioural data — actual usage patterns, login cadence, feature adoption depth — would sharpen and validate these findings. This analysis is a strong starting point; a live deployment would layer in behavioural event data to improve signal quality.
